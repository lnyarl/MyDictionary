# Stashy 아키텍처 개요

## 프로젝트 개요

Stashy는 현대적인 마이크로서비스에서 영감을 받은 아키텍처로 설계된 풀스택 사전 애플리케이션입니다. 이 시스템은 백엔드, 프론트엔드, 관리자 백엔드, 관리자 프론트엔드 및 공유 워크스페이스를 명확하게 구분하여 관심사의 분리를 염두에 두고 구축되었습니다.

## 핵심 아키텍처 결정

### 1. 워크스페이스 분리를 통한 모노레포 구조

**이유**: 다음과 같은 목적으로 워크스페이스가 분리된 모노레포를 선택했습니다:
- 사용자 대면 기능과 관리자 기능 간의 명확한 분리 유지
- 공통 타입, DTO, 유틸리티를 결합 없이 공유
- 각 부분의 독립적인 배포 및 확장 가능
- 의존성 관리 및 버전 관리 간소화

**장점**:
- **격리**: 관리자 및 사용자 시스템은 독립적으로 발전할 수 있습니다
- **공유 코드**: 공통 타입 및 유효성 검사 로직이 한 곳에 있습니다
- **개발 효율성**: 일관된 도구를 사용하는 단일 저장소
- **보안**: 별도의 백엔드를 통해 다양한 보안 요구 사항을 수용할 수 있습니다

### 2. 이중 백엔드 아키텍처

**이유**: 메인 애플리케이션과 관리자 기능을 위한 별도의 백엔드를 사용하는 이유는 다음과 같습니다:
- **보안**: 관리자 기능은 다른 보안 모델과 권한을 필요로 합니다
- **성능**: 관리자 작업(대량 작업, 분석)이 사용자 경험에 영향을 미치지 않습니다
- **유지보수성**: 관심사의 명확한 분리로 코드베이스를 이해하기 쉽습니다
- **확장성**: 관리자 및 사용자 트래픽을 독립적으로 확장할 수 있습니다

### 3. Radix UI 기반 React 19

**이유**: 프론트엔드에 React 19와 Radix UI를 선택한 이유는 다음과 같습니다:
- **접근성**: Radix UI는 기본적으로 접근성 높은 프리미티브를 제공합니다
- **스타일 없는 컴포넌트**: Tailwind로 커스텀 디자인 시스템을 자유롭게 구축할 수 있습니다
- **최신 기능**: 더 나은 개발자 경험을 위한 React 19의 최신 기능
- **성능**: 빠른 개발 및 최적화된 빌드를 위한 Vite

### 4. NestJS + Knex.js 백엔드 스택

**이유**: 이 조합은 다음을 위해 선택되었습니다:
- **타입 안전성**: 프론트엔드에서 데이터베이스까지 전체 TypeScript 스택
- **데이터베이스 유연성**: Knex.js는 ORM 제한 없이 SQL 쿼리 빌더의 유연성을 제공합니다
- **엔터프라이즈 패턴**: NestJS는 검증된 엔터프라이즈 패턴 및 의존성 주입을 제공합니다
- **테스트 용이성**: 내장된 테스트 유틸리티 및 패턴

### 5. Redis 캐시를 사용한 PostgreSQL

**이유**: 
- **PostgreSQL**: 강력한 일관성 및 복잡한 쿼리 지원을 제공하는 견고한 관계형 데이터베이스
- **Redis**: 세션, 속도 제한 및 자주 액세스되는 데이터에 대한 고성능 캐싱

## 기술 스택 요약

### 프론트엔드 애플리케이션
- **코어**: React 19, TypeScript, Vite
- **UI**: Radix UI 프리미티브, Tailwind CSS
- **라우팅**: React Router v7
- **상태**: React Context + Hooks (외부 상태 관리 없음)
- **리치 텍스트**: TipTap 에디터
- **국제화**: i18next

### 백엔드 애플리케이션
- **코어**: NestJS, TypeScript
- **데이터베이스**: Knex.js 쿼리 빌더, PostgreSQL
- **인증**: JWT 및 Google OAuth를 사용한 Passport.js
- **유효성 검사**: class-validator 및 class-transformer
- **파일 스토리지**: AWS S3 SDK
- **캐시**: Redis (ioredis)

### 공유 워크스페이스
- **목적**: 공통 타입, DTO, 상수, 유틸리티
- **배포**: 원활한 공유를 위한 Yarn 워크스페이스

### 개발 도구
- **린팅/포맷팅**: Biome (ESLint + Prettier 대체)
- **테스트**: 단위 테스트용 Jest, E2E용 Playwright
- **컨테이너화**: docker-compose를 사용한 Docker
- **Git Hooks**: lint-staged를 사용한 Husky

## 주요 아키텍처 패턴

### 1. 기본 리포지토리가 있는 리포지토리 패턴

모든 데이터베이스 액세스는 공통 `BaseRepository`를 확장하는 리포지토리를 통해 이루어집니다. 이는 다음을 제공합니다:
- 일관된 데이터베이스 액세스 패턴
- 내장된 소프트 삭제 기능
- 트랜잭션 지원
- 쿼리 빌딩 추상화

### 2. DTO 기반 API 유효성 검사

모든 API 엔드포인트는 다음을 위해 class-validator 데코레이터가 있는 DTO를 사용합니다:
- 입력 유효성 검사 및 정화
- 자동 타입 변환
- 명확한 API 계약
- 프론트엔드와 백엔드 간의 공유 유효성 검사 로직

### 3. 전반적인 의존성 주입

NestJS의 의존성 주입은 다음을 위해 광범위하게 사용됩니다:
- 서비스 구성
- 구성 관리
- 테스트 더블 및 모의
- 관심사의 명확한 분리

### 4. 모듈식 설계

프론트엔드와 백엔드 모두 기능 모듈로 구성됩니다:
- 사용자 관리
- 사전 콘텐츠
- 인증
- 관리자 작업
- 공유 유틸리티

## 데이터 흐름 아키텍처

```
사용자 요청 → 프론트엔드 → API 호출 → 백엔드 → 리포지토리 → 데이터베이스
                       ↓
                  유효성 검사 및 인증
                       ↓
                  비즈니스 로직
                       ↓
                  데이터베이스 작업
                       ↓
                  응답 및 캐싱
```

## 보안 아키텍처

### 인증 흐름
1.  **Google OAuth**: 기본 인증 방법
2.  **JWT 토큰**: 스테이트리스(Stateless) 세션 관리
3.  **갱신 토큰**: 보안 토큰 갱신
4.  **역할 기반 액세스**: 사용자 및 관리자에 대한 다른 권한

### 데이터 보호
- **입력 유효성 검사**: DTO를 통해 모든 입력 유효성 검사
- **SQL 주입 방지**: Knex.js 매개변수화된 쿼리
- **XSS 보호**: 콘텐츠 보안 정책 및 입력 정화
- **속도 제한**: Redis 기반 속도 제한

## 개발 철학

1.  **타입 안전성 우선**: Strict 모드를 사용한 전체 TypeScript 적용
2.  **개발자 경험**: 핫 리로딩, 포괄적인 린팅, 빠른 빌드
3.  **테스트 용이성**: 내장된 테스트 패턴 및 유틸리티
4.  **일관성**: 모든 코드에서 공유되는 패턴 및 컨벤션
5.  **성능**: 개발 및 프로덕션 모두에 최적화
