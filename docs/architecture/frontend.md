# 프론트엔드 아키텍처

## 개요

Stashy 프론트엔드는 **React 19**와 **TypeScript**로 구축된 현대적인 웹 애플리케이션이며, 빠른 개발 경험을 위해 **Vite**를 활용합니다. 접근성을 고려한 스타일 없는 컴포넌트를 위해 **Radix UI**를 사용하며, **Tailwind CSS**로 커스텀 디자인 시스템을 구축할 수 있도록 합니다. **React Router v7**은 클라이언트 측 라우팅을 처리하여 원활한 사용자 경험을 제공합니다.

## 기술 스택

### 핵심 프레임워크
- **React 19**: 대화형 사용자 인터페이스 구축을 위한 UI 라이브러리
- **TypeScript**: 타입 안전성을 보장하고 코드 품질을 향상
- **Vite**: 빠른 개발 및 최적화된 빌드를 위한 차세대 프론트엔드 도구

### UI 및 스타일링
- **Radix UI**: 고품질 디자인 시스템 구축을 위한 스타일 없는, 접근성 있는 컴포넌트
- **Tailwind CSS**: 빠른 UI 개발을 위한 유틸리티 우선 CSS 프레임워크
- **class-variance-authority & clsx**: 컴포넌트 변형 및 조건부 클래스 관리를 위한 유틸리티

### 라우팅 및 상태 관리
- **React Router v7**: 탐색을 위한 선언적 라우팅
- **React Context + Hooks**: 상태 관리를 위한 React 네이티브 기능 (외부 상태 관리 라이브러리 없음)
- **react-use**: 유용한 React Hooks 모음

### 기타 라이브러리
- **TipTap**: 콘텐츠 생성을 위한 확장 가능한 리치 텍스트 편집기
- **i18next & react-i18next**: 다국어 지원을 위한 국제화
- **lucide-react**: 아름답게 제작된 오픈 소스 아이콘
- **uuid**: 고유 ID 생성을 위해

## 아키텍처 패턴

### 1. 컴포넌트 기반 아키텍처

프론트엔드는 UI를 재사용 가능하고 모듈식 컴포넌트로 구성하는 컴포넌트 기반 아키텍처를 따릅니다. 컴포넌트는 일반적으로 `frontend/src/components/` 및 `frontend/src/pages/`에 위치합니다.

```
frontend/src/
├── components/              # 재사용 가능한 UI 컴포넌트
│   ├── ui/                  # Radix UI + Tailwind 스타일링된 프리미티브 (shadcn/ui-와 유사)
│   ├── layout/              # 레이아웃 컴포넌트 (헤더, 푸터, 사이드바)
│   └── specific/            # 기능별 컴포넌트
├── pages/                   # 최상위 페이지 컴포넌트 (DashboardPage, SettingsPage)
├── lib/                     # 유틸리티 함수, 헬퍼, 상수
├── hooks/                   # 재사용 가능한 로직을 위한 커스텀 React Hooks
├── context/                 # 전역 상태를 위한 React Context 제공자
├── routes/                  # 경로 정의 및 레이아웃
├── App.tsx                  # 메인 애플리케이션 컴포넌트
└── main.tsx                 # 애플리케이션 진입점
```

**이 패턴을 사용하는 이유**: 컴포넌트 기반 아키텍처는 다음을 제공합니다:
- **재사용성**: 애플리케이션의 여러 부분에서 컴포넌트를 사용할 수 있습니다
- **유지보수성**: UI의 격리된 부분을 관리하고 업데이트하기 쉽습니다
- **테스트 용이성**: 개별 컴포넌트를 독립적으로 테스트할 수 있습니다
- **확장성**: 대규모 애플리케이션 개발을 용이하게 합니다

### 2. 커스텀 스타일링을 통한 UI 프리미티브

이 프로젝트는 Radix UI의 접근성 있는 스타일 없는 프리미티브를 Tailwind CSS와 함께 사용하여 스타일링합니다. 이 접근 방식을 통해 Radix UI의 접근성 기능을 활용하면서 시각적 디자인을 완전히 제어할 수 있습니다.

**이 접근 방식을 사용하는 이유**:
- **접근성**: Radix UI는 기본적으로 높은 접근성 표준을 보장합니다
- **유연성**: Tailwind CSS로 컴포넌트를 자유롭게 스타일링할 수 있습니다
- **성능**: 필요한 CSS만 번들됩니다
- **일관성**: 강력한 프리미티브를 기반으로 구축된 중앙 집중식 디자인 시스템

### 3. 전역 상태 관리를 위한 React Context

전역 상태는 React의 Context API와 `useReducer` 또는 `useState` 훅을 사용하여 관리됩니다. 이는 더 간단한 전역 상태 요구 사항에 대해 Redux 또는 Zustand와 같은 외부 상태 관리 라이브러리의 필요성을 피합니다.

**이 접근 방식을 사용하는 이유**:
- **단순성**: 내장된 React 기능을 활용합니다
- **번들 크기 감소**: 추가 라이브러리 오버헤드 없음
- **명확성**: 상태 관리가 종종 이를 사용하는 컴포넌트와 함께 위치합니다

### 4. 탐색을 위한 React Router

React Router v7은 애플리케이션 경로를 정의하고 관리하는 데 사용됩니다. 이는 클라이언트 측 라우팅을 가능하게 하여 단일 페이지 애플리케이션 경험을 제공합니다.

**이 접근 방식을 사용하는 이유**:
- **SPA 경험**: 페이지 간 빠르고 부드러운 전환
- **선언적 라우팅**: 경로가 명확하고 예측 가능하게 정의됩니다
- **중첩 라우팅**: 중첩된 경로로 복잡한 UI 레이아웃 지원

## 주요 구성 요소

### 1. 레이아웃 컴포넌트

레이아웃(예: `AuthLayout`, `MainLayout`)은 페이지를 래핑하고 헤더, 푸터 및 탐색과 같은 공통 UI 요소를 제공하는 데 사용됩니다. 이는 애플리케이션 전체에서 일관된 모양과 느낌을 보장합니다.

### 2. UI 컴포넌트 (`components/ui`)

이 디렉토리에는 Radix UI 프리미티브를 사용하여 구축되고 Tailwind CSS로 스타일링된 일반적이고 재사용 가능한 UI 컴포넌트가 포함되어 있습니다. 예시로는 `Button`, `Input`, `Dialog`, `DropdownMenu` 등이 있습니다.

### 3. 페이지 (`pages/`)

페이지 컴포넌트는 애플리케이션의 다른 뷰 또는 화면(예: `DashboardPage`, `SettingsPage`, `WordDetailPage`)을 나타냅니다. 일반적으로 여러 개의 작은 컴포넌트를 구성하여 완전한 UI를 형성합니다.

### 4. 커스텀 훅 (`hooks/`)

커스텀 훅은 재사용 가능한 로직과 상태 저장 동작을 캡슐화하는 데 사용됩니다. 이는 코드 공유를 촉진하고 중복을 줄입니다.

## 개발 워크플로우

### 린팅 및 포맷팅
- **Biome**: 일관된 코드 스타일을 보장하기 위해 린팅 및 포맷팅에 사용됩니다.
- **들여쓰기**: 프론트엔드 파일에서는 탭이 들여쓰기에 사용됩니다.

### 타입 검사
- **TypeScript**: `tsc -b`는 타입 관련 오류를 조기에 감지하기 위해 타입 검사에 사용됩니다.

### 빌드
- **Vite**: 최적화된 프로덕션 빌드를 생성하기 위한 `vite build` 명령.

## 성능 고려 사항

- **Vite**: 빠른 개발 서버 및 최적화된 프로덕션 빌드 (트리 쉐이킹, 코드 분할).
- **Radix UI**: 성능 및 접근성에 고도로 최적화되었습니다.
- **이미지 최적화**: 프로덕션에서 외부 도구 또는 서비스로 처리될 가능성이 높습니다.
- **지연 로딩**: 초기 번들 크기를 줄이기 위해 경로 및 대형 컴포넌트에 사용됩니다.

## 국제화

- **i18next**: 다국어 지원 처리를 위해.
- **언어 감지**: 자동 언어 감지를 위한 `i18next-browser-languagedetector`.
- **HTTP 백엔드**: 비동기적으로 번역 파일을 로드하기 위한 `i18next-http-backend`.
