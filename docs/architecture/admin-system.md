# 관리 시스템 아키텍처

## 개요

Stashy 애플리케이션은 별도의 백엔드(`backend-admin`) 및 프론트엔드(`frontend-admin`)로 구성된 전용 관리 시스템을 특징으로 합니다. 이러한 분리는 관리 작업을 위한 보안, 유지보수성 및 확장성을 향상시키기 위한 핵심 아키텍처 결정입니다.

## 아키텍처 근거

**별도의 관리 시스템을 사용하는 이유?**

1.  **보안**: 관리 기능은 종종 일반 사용자 대면 애플리케이션과 비교하여 다른 인증 메커니즘, 권한 수준 및 노출된 API를 필요로 합니다. 이를 분리하면 공격 표면을 최소화하고 관리 작업을 위한 더 엄격한 보안 프로토콜을 허용합니다.
2.  **성능 및 안정성**: 관리 작업(예: 대량 데이터 작업, 보고, 사용자 관리)은 리소스 집약적일 수 있습니다. 이를 격리하면 일반 사용자를 위한 잠재적인 성능 저하 또는 안정성 문제를 방지할 수 있습니다.
3.  **유지보수성 및 복잡성**: 관리 기능은 본질적으로 다른 비즈니스 로직 및 UI 요구 사항을 가집니다. 별도의 시스템은 메인 애플리케이션 또는 관리자 패널에서 작업하는 개발자의 인지 부하를 줄여 더 깨끗한 코드베이스와 더 쉬운 유지보수를 가능하게 합니다.
4.  **확장성**: 메인 애플리케이션과 관리 시스템은 각각의 트래픽 및 리소스 요구 사항에 따라 독립적으로 확장될 수 있습니다.

## 관리 백엔드 아키텍처 (`backend-admin`)

`backend-admin`은 메인 백엔드(`backend`)의 핵심 아키텍처를 크게 따르지만, 관리 기능에 맞게 조정됩니다.

### 기술 스택
-   **코어**: NestJS, TypeScript
-   **데이터베이스**: Knex.js (SQL 쿼리 빌더), PostgreSQL
-   **인증**: JWT 전략을 사용한 Passport.js, 암호 해싱을 위한 `bcrypt` (로컬 관리자 계정용)
-   **유효성 검사**: `class-validator` 및 `class-transformer`
-   **공유 타입**: 공통 DTO 및 인터페이스를 위한 `@stashy/shared` 워크스페이스

### 주요 차이점 및 세부 사항

1.  **인증**: 두 백엔드 모두 JWT를 사용하지만, 관리 백엔드는 별도의 인증 흐름을 구현할 가능성이 높습니다. `bcrypt`를 의존성으로 포함하는데, 이는 Google과 같은 외부 OAuth 제공자에만 의존하기보다는 로컬 관리자 사용자 자격 증명(사용자 이름/암호) 관리를 시사합니다.
2.  **API 엔드포인트**: 관리 백엔드는 다음을 위한 고유한 API 세트를 노출합니다:
    -   사용자 관리 (예: 사용자 정지, 역할 변경, 계정 삭제)
    -   콘텐츠 조정 (예: 정의 승인/거부, 단어 관리)
    -   시스템 구성 및 설정
    -   분석 및 보고 도구
3.  **데이터베이스 상호 작용**: 메인 애플리케이션과 동일한 PostgreSQL 데이터베이스와 상호 작용하지만, 종종 다른 유형의 작업(예: 직접 데이터 조작, 대량 업데이트, 보고를 위한 복잡한 쿼리)을 수행합니다.
4.  **미들웨어 및 가드**: 관리자 역할 및 권한에 특정한 커스텀 NestJS 가드 및 미들웨어를 포함할 수 있으며, 승인된 관리자만 민감한 엔드포인트에 액세스할 수 있도록 합니다.

## 관리 프론트엔드 아키텍처 (`frontend-admin`)

`frontend-admin`은 메인 프론트엔드와 유사한 기술로 구축되었지만, 관리자 사용자 인터페이스에 중점을 둔 React 애플리케이션입니다.

### 기술 스택
-   **코어**: React 19, TypeScript, Vite
-   **UI**: Radix UI (접근성 있는 프리미티브), Tailwind CSS
-   **라우팅**: React Router v7
-   **공유 타입**: 공통 타입 및 DTO를 위한 `@stashy/shared` 워크스페이스

### 주요 차이점 및 세부 사항

1.  **사용자 인터페이스**: 관리 작업을 위해 특별히 설계되었으며, 다음을 특징으로 합니다:
    -   주요 메트릭을 포함하는 대시보드
    -   사용자, 단어, 정의, 보고서 목록을 위한 테이블
    -   사용자 역할, 콘텐츠 속성, 시스템 설정 편집을 위한 양식
    -   콘텐츠 조정 도구
2.  **컴포넌트 세트**: 기본 UI 컴포넌트(Radix UI 및 Tailwind)를 재사용하지만, 도메인별 컴포넌트(예: `UserTable`, `ReportApprovalCard`, `SettingsForm`)를 가집니다.
3.  **상태 관리**: 메인 프론트엔드와 유사하게, 상태 관리를 위해 React Context 및 훅을 사용하지만, 관리자 특정 데이터 및 워크플로우에 맞게 조정됩니다.
4.  **라우팅**: 관리자 섹션을 탐색하기 위한 경로가 정의되며, 인증된 관리자만 액세스할 수 있도록 인증 검사에 의해 보호됩니다.

## 공유 워크스페이스와의 상호 작용

`backend-admin`과 `frontend-admin` 모두 `@stashy/shared` 워크스페이스를 활용합니다. 이는 다음을 위해 중요합니다:
-   **타입 일관성**: 관리 백엔드와 프론트엔드 간에 일관된 데이터 구조(DTO, 인터페이스)를 보장합니다.
-   **코드 재사용성**: 관리자 및 일반 사용자 대면 애플리케이션 부분 모두에 관련되는 유효성 검사 스키마, 상수 및 유틸리티 함수를 공유합니다.

## 관리 시스템을 위한 인증 흐름

일반적으로 관리자 인증 흐름은 다음을 포함합니다:
1.  **관리자 로그인**: 관리자는 전용 관리자 로그인 페이지를 통해 로그인하며, 종종 로컬 자격 증명을 사용합니다.
2.  **JWT 발행**: 성공적인 인증 후, `backend-admin`은 세션 관리를 위해 JWT를 발행합니다.
3.  **역할 기반 액세스**: JWT에는 `backend-admin`이 특정 관리자 API 엔드포인트에 대한 액세스를 승인하는 데 사용하는 역할 정보가 포함됩니다.
4.  **프론트엔드 보호**: `frontend-admin`은 경로를 보호하기 위해 라우팅 가드 또는 컨텍스트 제공자를 사용하며, 인증되지 않은 사용자를 로그인 페이지로 리디렉션합니다.

이 아키텍처는 Stashy 애플리케이션 내에서 관리 기능을 관리하기 위한 안전하고 효율적이며 잘 조직된 접근 방식을 보장합니다.
