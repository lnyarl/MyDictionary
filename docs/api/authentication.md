# 인증 및 권한 부여 패턴

## 개요

Stashy 애플리케이션은 주요 사용자 대면 애플리케이션과 전용 관리 패널 전반에 걸쳐 강력한 인증 및 권한 부여 시스템을 구현합니다. 두 시스템 모두 스테이트리스(Stateless) 세션 관리를 위해 JSON 웹 토큰(JWT)을 활용하지만, 특정 보안 요구 사항에 맞춰 고유한 흐름과 메커니즘을 사용합니다.

## 1. 사용자 대면 애플리케이션 인증 (백엔드 및 프론트엔드)

### 백엔드 (`backend`)

#### 기술 스택
-   **Passport.js**: 인증 미들웨어
-   **passport-jwt**: Passport.js용 JWT 전략
-   **google-auth-library**: Google ID 토큰 확인용
-   **@nestjs/jwt**: JWT 관리를 위한 NestJS 모듈
-   **cookie-parser**: HTTP 전용 쿠키의 `access_token` 처리를 위해

#### 인증 흐름

1.  **Google OAuth 로그인 (`POST /auth/google`)**:
    -   프론트엔드가 Google ID 토큰(자격 증명)을 백엔드로 보냅니다.
    -   `AuthService`는 `google-auth-library`로 ID 토큰을 확인하여 사용자 데이터(Google ID, 이메일, 이름, 사진)를 추출합니다.
    -   `AuthService`는 `googleId`를 가진 사용자가 이미 존재하는지 확인합니다. 존재하지 않으면 생성된 임의의 닉네임으로 `users` 테이블에 새 사용자 계정을 생성합니다.
    -   유효성 검사/생성이 성공하면 `AuthService`는 `@nestjs/jwt`를 사용하여 JWT를 생성합니다.
    -   **JWT 및 Google OAuth를 사용하는 이유**: JWT는 스테이트리스하고 확장 가능한 인증 메커니즘을 제공하며, Google OAuth는 자격 증명 관리를 오프로드하여 사용자 등록 및 로그인을 간소화하고 사용자 경험과 보안을 향상시킵니다.
2.  **JWT 발행 및 쿠키 관리**:
    -   생성된 JWT(`access_token`)는 클라이언트 측에 **HTTP 전용 쿠키**로 설정됩니다.
    -   **HTTP 전용 쿠키를 사용하는 이유**: 이는 클라이언트 측 JavaScript가 토큰에 액세스하는 것을 방지하여 XSS(교차 사이트 스크립팅) 공격을 완화합니다. `secure: !isDevelopment` 및 `sameSite: isDevelopment ? "lax" : "none"` 설정은 개발 및 프로덕션 환경 전반에 걸쳐 올바른 동작과 프로덕션 환경에서 HTTPS를 통한 보안 전송을 보장합니다.
    -   토큰은 일반적으로 짧은 만료 기간(예: 7일)을 가집니다.
3.  **세션 생성 (`POST /auth/session`)**:
    -   이 엔드포인트를 사용하면 프론트엔드가 제공된 토큰을 HTTP 전용 쿠키에 명시적으로 설정할 수 있습니다. 이는 외부 소스에서 토큰을 얻거나 갱신하여 기본 세션 토큰으로 설정해야 하는 시나리오에 유용합니다.
    -   **이유**: 특히 서버 측 렌더링 페이지 또는 초기 로드 시나리오를 처리할 때 클라이언트 측에서 쿠키의 `access_token`을 관리할 수 있는 유연성을 제공합니다.

#### 권한 부여 (가드)

1.  **`JwtAuthGuard`**:
    -   **목적**: 유효한 JWT를 필요로 하는 API 엔드포인트를 보호합니다.
    -   **메커니즘**: `AuthGuard('jwt')`를 확장합니다. `access_token` HTTP 전용 쿠키에서 JWT를 추출합니다.
    -   **유효성 검사**: `JwtStrategy`는 토큰을 유효성 검사하고 페이로드에서 `sub` (사용자 ID) 및 `email`을 추출하며, 데이터베이스에서 해당 사용자를 가져옵니다(`UsersService`).
    -   **`@Public()` 데코레이터**: `@Public()`으로 데코레이터된 엔드포인트(예: `/auth/google`)는 `JwtAuthGuard`를 우회하여 인증되지 않은 액세스를 허용합니다.
    -   **이유**: 인증된 사용자만 보호된 리소스에 액세스할 수 있도록 보장하여 첫 번째 방어 계층을 제공합니다.
2.  **`SuspendedUserGuard`**:
    -   **목적**: 정지된 사용자가 쓰기 작업을 수행하는 것을 방지합니다.
    -   **메커니즘**: 인증된 사용자의 `suspendedAt` 필드를 확인합니다.
    -   **동작**: `GET` 요청(읽기 전용 작업)은 허용하지만, 사용자가 정지된 경우 `POST`, `PUT`, `PATCH`, `DELETE` 요청에 대해 `ForbiddenException`을 발생시킵니다.
    -   **이유**: 관리자가 문제가 있는 사용자의 작업을 제한하면서도 콘텐츠를 탐색할 수 있도록 허용하는 데 중요합니다.

#### 사용자 세션 관리
-   **`GET /auth/me`**: 현재 인증된 사용자의 프로필을 검색합니다.
-   **`POST /auth/logout`**: `access_token` 쿠키를 지워 사용자를 효과적으로 로그아웃시킵니다.

### 프론트엔드 (`frontend`)

-   **인증 상태**: 프론트엔드는 React Context 및 커스텀 훅을 사용하여 사용자 인증 상태(예: `isLoggedIn`, `userProfile`)를 관리합니다.
-   **보호된 경로**: `AuthLayout` 또는 유사한 메커니즘을 사용하여 인증이 필요한 경로를 래핑합니다. 사용자가 인증되지 않은 상태에서 보호된 경로에 액세스하려고 하면 로그인 페이지로 리디렉션됩니다.
-   **API 상호 작용**: 프론트엔드는 (HTTP 전용 쿠키에 대해 브라우저가 관리하는) 쿠키를 통해 요청과 함께 `access_token`을 자동으로 보냅니다.

## 2. 관리 시스템 인증 (관리자 백엔드 및 프론트엔드)

### 관리자 백엔드 (`backend-admin`)

#### 기술 스택
-   **Passport.js**: 인증 미들웨어
-   **passport-jwt**: Passport.js용 JWT 전략
-   **bcrypt**: 관리자 암호 해싱 및 비교용
-   **@nestjs/jwt**: JWT 관리를 위한 NestJS 모듈
-   **cookie-parser**: HTTP 전용 쿠키의 `admin_access_token` 처리를 위해

#### 인증 흐름

1.  **로컬 관리자 로그인 (`POST /auth/login`)**:
    -   관리자는 사용자 이름과 암호(로컬 자격 증명, OAuth 아님)를 사용하여 로그인합니다.
    -   `AuthService`는 `admin_users` 테이블에 대해 이러한 자격 증명을 유효성 검사합니다.
    -   `bcrypt`는 암호를 안전하게 해싱하고 비교하는 데 사용됩니다.
    -   **왜 로컬 자격 증명인가**: 관리 직원에게 공용 사용자 계정과 별도로 전용의 격리된 인증 메커니즘을 제공하여 더 엄격한 제어 및 규정 준수를 가능하게 합니다.
2.  **JWT 발행 및 쿠키 관리**:
    -   로그인 성공 시 JWT(`admin_access_token`)가 생성되어 **HTTP 전용 쿠키**로 설정됩니다.
    -   **HTTP 전용 쿠키를 사용하는 이유**: 메인 애플리케이션과 동일한 보안 이점을 제공합니다.
    -   관리자 JWT는 일반적으로 더 짧은 만료 기간(예: 8시간)을 가집니다.

#### 권한 부여 (가드 및 역할 기반 접근 제어 - RBAC)

1.  **`AdminJwtAuthGuard`**:
    -   **목적**: 관리자 API 엔드포인트를 보호합니다.
    -   **메커니즘**: `JwtAuthGuard`와 유사하지만, 쿠키에서 `admin_access_token`을 추출하고 `AdminJwtStrategy`를 사용합니다.
    -   **유효성 검사**: `AdminJwtStrategy`는 토큰을 유효성 검사하고 `admin_users` 테이블에서 해당 `admin_user`를 가져옵니다.
    -   **`@Public()` 데코레이터**: 특정 관리자 엔드포인트(예: `/auth/login`)가 이 가드를 우회할 수 있도록 허용합니다.
    -   **이유**: 인증된 관리자만 관리자 기능에 액세스할 수 있도록 보장합니다.
2.  **`RolesGuard`**:
    -   **목적**: 세분화된 권한을 위한 역할 기반 접근 제어(RBAC)를 구현합니다.
    -   **메커니즘**: `@Roles()` 데코레이터(예: `@Roles(AdminRole.SUPER_ADMIN, AdminRole.OPERATOR)`)를 사용하여 특정 엔드포인트에 액세스할 수 있는 역할을 지정합니다.
    -   **동작**: 인증된 `AdminUser`의 `role` 필드를 필요한 역할과 비교합니다.
    -   **`AdminRole.SUPER_ADMIN`**: 항상 역할 검사를 우회합니다.
    -   **이유**: 관리 작업에 대한 세분화된 제어를 제공하여 적절한 권한을 가진 사용자만 민감한 작업을 수행할 수 있도록 보장합니다.
3.  **`PasswordChangeRequiredGuard`**:
    -   **목적**: 새 또는 재설정된 관리자 계정에 대한 암호 변경을 강제합니다.
    -   **메커니즘**: `AdminUser`의 `mustChangePassword` 플래그를 확인합니다.
    -   **`@SkipPasswordCheck()` 데코레이터**: 특정 엔드포인트(예: `GET /auth/me` 또는 `POST /auth/change-password`)가 이 가드를 우회할 수 있도록 허용합니다.
    -   **이유**: 관리자가 초기 로그인 시 또는 암호 재설정 후 보안 암호를 설정하도록 요구하여 보안을 강화합니다.

#### 관리자 세션 관리
-   **`GET /auth/me`**: 현재 인증된 관리자의 프로필을 검색합니다.
-   **`POST /auth/logout`**: `admin_access_token` 쿠키를 지웁니다.
-   **`POST /auth/change-password`**: 관리자가 암호를 변경할 수 있도록 하며, 이는 또한 `mustChangePassword` 플래그를 지웁니다.

### 관리자 프론트엔드 (`frontend-admin`)

-   **인증 상태**: 관리자의 로그인 상태 및 역할 정보를 관리합니다.
-   **보호된 경로**: 관리자별 라우팅은 인증되지 않은 사용자가 관리자 로그인 페이지로 리디렉션되도록 보장합니다. 역할 기반 검사는 관리자의 권한에 따라 UI 요소를 숨기거나 표시하기 위해 클라이언트 측에서도 구현될 수 있습니다.

## 공유 인증 로직 및 DTO

-   **`@stashy/shared`**: (메인 앱용 `GoogleLoginDto`, 관리자용 `LoginDto`, `ChangePasswordDto`와 같은) 공통 인증 관련 DTO 및 프론트엔드 및 백엔드 시스템 모두에서 사용되는 인터페이스를 포함합니다.
-   **왜 공유하는가**: 인증 요청 및 응답에 대한 타입 일관성 및 명확한 API 계약을 보장합니다.

## 보안 고려 사항

-   **HTTP 전용 쿠키**: XSS 공격이 JWT에 액세스하는 것을 방지합니다.
-   **Secure & SameSite 플래그**: 쿠키가 안전하게 전송되도록 보장하고 CSRF(교차 사이트 요청 위조) 공격을 완화합니다.
-   **JWT 만료**: 단기 토큰은 토큰 침해 가능성을 줄입니다.
-   **암호 해싱**: `bcrypt`는 관리자 백엔드에서 안전한 암호 저장에 사용됩니다.
-   **역할 기반 접근 제어**: 사용자 역할에 따라 민감한 기능에 대한 액세스를 제한합니다.
-   **정지된 사용자**: 악의적이거나 문제가 있는 사용자가 작업을 수행하는 것을 방지합니다.
-   **입력 유효성 검사**: 모든 인증 관련 입력은 DTO를 통해 유효성 검사됩니다.

---

이 포괄적인 인증 및 권한 부여 전략은 Stashy 애플리케이션 내에서 사용자 및 관리자 액세스를 관리하기 위한 안전하고 유연한 프레임워크를 제공합니다.
